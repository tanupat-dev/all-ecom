# `shipping_overcharge` is detected per-platform from each Platform's *disclosed actual* shipping (subsidy-aware) — catalogue-compute (ADR 0022) is demoted to a fallback

---
Status: accepted — **supersedes ADR 0022's *primary* mechanism**. ADR 0022 (expected shipping computed from catalogue weight/dimensions, compared against `|shipping_seller_paid|`) is **demoted to the last-resort fallback** for a Platform/Shop that exposes no per-order actual shipping. The uniform "one formula, all platforms, net-`ShippingSellerPaid` vs catalogue" flag it defined is **no longer the primary path** — it false-flags subsidised sellers and misses real overcharges (see Why).
---

Phase 8's `shipping_overcharge` **Claim** (CONTEXT.md: Claim) auto-flags when a courier billed the seller more than the parcel's true weight warrants. ADR 0022 pinned a single catalogue-derived expectation compared against the netted `ShippingSellerPaid` Accounting line for **all three** Platforms. Reading the **real export files** (`ref doc/`, the authoritative `/mnt/c` copies) order-by-order showed that single shape is wrong for every Platform: each one **discloses the actual charged shipping itself** — in a *different file, at a different granularity, with different subsidy structure* — and the netted-vs-catalogue comparison either false-flags or misses the real overcharge. This ADR pins a **per-platform, subsidy-aware** detection grounded in those columns.

## What the real files disclose (verified order-by-order)

- **TikTok — `Accounting tiktok.xlsx`, per order.** Carries **both weights**: `น้ำหนักพัสดุโดยประมาณ` (parcel/declared weight) and `น้ำหนักที่เรียกเก็บเงิน` (the weight the courier billed), plus the **gross** `ค่าธรรมเนียมการจัดส่งจริง` separate from `ส่วนลดค่าธรรมเนียมการจัดส่งจากแพลตฟอร์ม` (a platform discount). The importer currently sums every shipping column into one `ShippingSellerPaid` bucket → the discount **masks** the gross overcharge (real example: gross fee ฿48 on a **1090 g** charged weight for a **380 g** parcel, discount +฿35, net **฿13** — below any expected rate, so the netted-vs-catalogue rule **never fires**). The two weights are dropped on import today.
- **Shopee — `Accounting shopee 2.xlsx` → "Income" sheet, per order.** No weight anywhere, but the actual charge is fully itemised: `ค่าจัดส่งที่ชำระโดยผู้ซื้อ` (buyer-paid) + `ค่าจัดส่งสินค้าที่ออกโดย Shopee` (subsidy) + `ค่าจัดส่งที่ Shopee ชำระโดยชื่อของคุณ` (actual courier charge, negative). For a **free-shipping-programme** seller the subsidy + buyer-paid **exactly offset** the actual charge → seller **net shipping = 0** (verified: 855 orders, Mar–Jun, per-order sum reconciles to the file's own Summary `ค่าจัดส่ง = 0`). A gross- or catalogue-based comparison would **false-flag every one** of those orders. The actual courier charge is **not** in the order export (estimates only) nor the wallet "Transaction Report" (net payout + return clawbacks only).
- **Lazada — `Shipping fee lazada.xlsx` (Finance › Shipping Fee / Logistics Billing), per *billing cycle*.** Discloses the overcharge directly: `Package qty of lsf less than csf` (Lazada-fee < charged-fee = overcharged packages), `Wrong weight adjustment`, `Shipping fee charged by lazada`, `Shipping fee voucher by lazada`. **Statement-level, not per-order**, and hard to export (cycle by cycle). The bulk `Accounting`/`All order` exports we already ingest carry only a buyer-facing `shippingFee` and a flat "free-shipping programme" fee — **no** weight, no per-order actual courier charge.

## Decision

1. **Detection is per-platform, against the Platform's own disclosed actual shipping — never a single uniform formula.**
   - **TikTok → weight-discrepancy.** Import the two weight columns + the **gross** `ค่าธรรมเนียมการจัดส่งจริง` as a distinct signal (not melted into the net `ShippingSellerPaid` bucket). Flag when the **billed weight materially exceeds the parcel/declared weight**. The recoverable is the **gross** overcharge, because the platform discount does not scale with the weight correction. (Net masks it — see Why.)
   - **Shopee → subsidy-aware net.** Source the **Income sheet** of the income report. Compute the **seller's net shipping per order** = `buyer-paid + subsidy + actual-courier-charge + return-shipping + savings`. Only an order with **net < 0** (the seller actually bore shipping) is a candidate; among those, flag when the borne amount exceeds the catalogue expectation (ADR 0022 fallback) or a configured rate. A fully-subsidised order nets to 0 → **no flag** (correct).
   - **Lazada → statement-level, from the Shipping Fee statement.** Flag from `Package qty of lsf less than csf` / `Wrong weight adjustment`. Because it is **per billing cycle, not per Order**, a Lazada `shipping_overcharge` attaches to the **statement/cycle** (or is created manually) rather than auto-attaching to one Order.
2. **Catalogue-compute (ADR 0022) is the fallback, not the primary.** When a Platform/Shop exposes no per-order actual shipping (or the seller does not provide the disclosed-actual export), fall back to `max(actual, volumetric)/5000` tiered against `expected_shipping_rate`. Missing inputs remain **fail-safe → no flag** (ADR 0005), exactly as ADR 0022 pinned.
3. **Subsidy-aware is mandatory.** The comparison is always against what the **seller actually bore**, never the gross courier charge in isolation — otherwise a subsidised seller is false-flagged on every order. (The one exception is TikTok, where the signal is the *weight* discrepancy, not the money, precisely because the money is netted down by a masking discount.)
4. **Recoverable = estimate + captured actual; we never hard-commit gross-vs-net.** The Claim page shows an **estimated** recoverable (`actual − expected@true-weight`) as a *worth-filing?* prompt, labelled an estimate. The **actual recovered** amount is read from the Platform's own compensation field on import — TikTok `Compensation Amount`, Shopee `เงินชดเชยให้ผู้ขาย` — into the Claim Timeline payout, rather than hand-typed. The platform is the source of truth for the won amount.
5. **Mirror native claim fields when the Platform discloses them.** TikTok return export carries `Dispute Status` / `Appeal Status` / `Compensation Status` (which map onto our two-stage `submitted_initial → submitted_ticket → approved/rejected` lifecycle); Shopee return export carries `สถานะการเคลมค่าจัดส่ง`, `ผู้ขายสามารถยื่นข้อพิพาทได้ภายใน` (the **filing deadline**), and `เงินชดเชยให้ผู้ขาย`. Where present, these **inform** Claim Status / deadline / payout on import; manual transition remains for stages the Platform does not cover and for seller-driven moves (e.g. `abandoned`).
6. **Manual Claim creation is allowed.** Not every overcharge is auto-flaggable (Lazada is statement-level; some are only visible to the seller's eye in Seller Centre). `ClaimResource::canCreate()` is opened (gated on `claim.manage`, ADR 0012) so a seller can raise a `shipping_overcharge` (or other dispute-portal category) by hand.
7. Money is integer **satang** throughout (ADR 0015).

## Why

- **Each Platform already tells us the answer; computing from catalogue is guessing when the source is in hand.** This mirrors the codebase posture (Expected Net from the Fee Profile, Available from the ledger, Effective Price from Promotion Lines): prefer the disclosed source of truth over a recomputation. ADR 0022's catalogue model was built before we read that the disclosed actuals exist per order.
- **Netting gross against a platform subsidy/discount is apples-to-oranges.** ADR 0022 compared a **gross** catalogue expectation against the **net** `ShippingSellerPaid` (which already has the platform subsidy subtracted). On Shopee that nets a fully-subsidised seller to 0 → comparing against a gross expectation false-flags; on TikTok the discount nets the ฿48 overcharge down to ฿13 → the overcharge vanishes below the expected rate and is missed. Only a **subsidy-aware** comparison (Shopee) or a **weight** comparison that ignores the money entirely (TikTok) survives the real data.
- **Multi-tenant, not single-shop.** This is a SaaS for many sellers. The example shop is on full free-shipping subsidy (net 0, nothing to claim) and is low-volume on Lazada — but other tenants will be non-subsidised on Shopee, high-volume on Lazada, and routinely weight-overcharged on TikTok. The per-platform rules are built for **all** of them; the subsidised tenant simply produces **0 flags** via the fail-safe, which is the rule working, not a reason to cut scope.
- **Estimate-plus-captured-actual dissolves the gross/net question.** We do not need to decide what fraction a Platform refunds: we *prompt* with an estimate and *record* whatever the Platform actually compensates, read from its own export.

## Consequences

- `FlagShippingOverchargeClaim` is no longer one uniform comparison — it becomes a **per-platform strategy** (TikTok weight-discrepancy · Shopee subsidy-aware net · Lazada statement-level), with `ComputeExpectedShipping` (ADR 0022) retained as the **fallback** expectation.
- **Two new importers**: Shopee income report ("Income" sheet, per-order shipping breakdown) and Lazada Shipping Fee statement (per billing cycle). Both must be read from the real files before building (ADR 0005 / project iron rule) — the columns above are the verified contract.
- The TikTok accounting importer must stop melting the weight columns and the gross shipping fee into a single `ShippingSellerPaid` bucket; it must retain the two weights and the gross fee for the weight-discrepancy signal (touches ADR 0020 line mapping).
- `ClaimResource::canCreate()` opens (gated `claim.manage`); the Resource gains a manual-create path and, for Lazada, a statement/cycle attachment rather than a single-Order one.
- Claim Status, filing deadline, and payout become **import-syncable** from native Platform fields where disclosed; the manual `TransitionClaimStatus` (Issue #81) remains the authority for uncovered stages and overrides.
- `expected_shipping_rate` (ADR 0022) survives unchanged as the fallback expectation; its Shop-Settings affordance is still wanted.
- A Lazada `shipping_overcharge` is **statement-scoped**, breaking the "a Claim attaches to exactly one Order" assumption for that one type — the model must allow a cycle/statement reference (or manual, Order-less, creation).

## Considered options

- **Keep ADR 0022 uniform catalogue-compute as primary.** Rejected — it false-flags every order of a subsidised Shopee seller and misses TikTok weight overcharges that the platform discount nets below the expected rate. Verified against real data, not hypothesised.
- **Net-based comparison for all three.** Rejected — TikTok's masking discount makes the seller's net *lower* than expected on a genuinely overcharged order; net would systematically miss TikTok.
- **Gross-based comparison for all three.** Rejected — false-flags subsidised sellers (Shopee net 0).
- **Auto-flag Lazada per Order from the bulk exports.** Rejected — the bulk `Accounting`/`All order` exports carry no per-order actual courier charge or weight; the disclosure lives only in the per-cycle Shipping Fee statement, which is statement-level and awkward to export. Statement-level / manual instead.
- **Hard-commit the recoverable to gross (or to net).** Rejected — the won amount is the Platform's to decide and it is disclosed on import; we estimate to prompt and capture the actual, avoiding a guess that would misstate money (ADR 0015 posture).
